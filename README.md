## nail-app – Telegram бот-записи для бьюти-мастеров

`nail-app` — это открытый Telegram-бот для индивидуальных бьюти-мастеров (изначально мастеров по маникюру), который помогает:

- принимать онлайн-записи от клиентов через личную ссылку бота;
- хранить базу клиентов с историей посещений;
- управлять услугами, расписанием и дневными лимитами по дням недели.

Бот ориентирован на **немного технических** мастеров и на **официально хостимую версию** для нетехнических пользователей.

---

## Для кого этот проект

- **Бьюти-мастера (маникюр, педикюр и т.д.)**, которые хотят быстро запустить личный бот для записи без отдельного сайта.
- **Разработчики**, которые хотят развернуть и доработать OSS-бота:
  - под своих клиентов;
  - под собственный сервис записи;
  - или как основу для продукта с расширенной аналитикой.

## MVP-функциональность

- **Роль Мастера**
  - Онбординг:
    - ввод имени мастера;
    - отправка прайс-листа (текст или фото, с рекомендацией текста);
    - парсинг прайса через LLM с валидацией и возможностью ручного редактирования;
    - настройка рабочих дней и окон (`Пн 10:00–12:00, 12:00–14:00` и т.д.);
    - настройка дневных лимитов по дням недели (`Пн – 3, Вт – 4` и т.д.);
    - генерация персональной ссылки вида `t.me/<bot_username>?start=master_<slug>`.
  - Управление услугами:
    - просмотр списка услуг;
    - добавление / редактирование / удаление;
    - изменение цены и длительности.
  - Управление расписанием:
    - рабочие дни недели;
    - окна по дням (`WorkingWindow`);
    - лимиты записей по дням (`DailyBookingLimit`).
  - Записи и клиенты:
    - просмотр ближайших записей по дням;
    - ручное добавление и отмена записей;
    - база клиентов (имя, Telegram, телефон, история).
  - Команда `/share`:
    - выдаёт личную ссылку на запись;
    - показывает готовый текст для отправки клиентам.

- **Роль Клиента**
  - Вход по ссылке мастера `t.me/<bot_username>?start=master_<slug>`.
  - Выбор услуги из списка с ценами.
  - Выбор дня:
    - учитываются только рабочие дни мастера;
    - исключаются дни, где достигнут дневной лимит;
    - исключаются дни без доступных окон под длительность услуги.
  - Выбор времени:
    - только слоты, которые полностью помещаются в рабочие окна;
    - без пересечений с существующими записями;
    - с учётом дневного лимита.
  - Подтверждение:
    - ввод имени;
    - опциональный телефон;
    - показ итоговой записи (услуга, дата, время, имя мастера).

Все пользовательские тексты в боте — **на русском языке**.

## Архитектура

Бот построен по слоистой архитектуре:

- **`bot`** — Telegram-уровень:
  - aiogram-хэндлеры;
  - клавиатуры;
  - тексты на русском;
  - сценарии онбординга мастера и записи клиента.
- **`services`** — прикладные сервисы (use cases):
  - оркестрация доменной логики, репозиториев и LLM-адаптеров;
  - онбординг мастера, управление услугами, расписанием, лимитами;
  - сценарий записи клиента.
- **`domain`** — чистая доменная логика:
  - сущности и value-объекты (Pydantic-модели);
  - генерация слотов, проверка пересечений;
  - применение дневных лимитов по дням недели;
  - схемы и валидация результатов LLM-парсинга.
- **`db`** — доступ к данным:
  - SQLAlchemy 2.x ORM-модели;
  - репозитории;
  - Alembic-миграции.
- **`adapters`** — интеграции и утилиты:
  - адаптеры к LLM (мок и опциональный OpenAI-совместимый);
  - абстракция `ImageToTextAdapter` (по умолчанию заглушка);
  - утилиты времени и часовых поясов.
- **`config`** — конфигурация:
  - Pydantic Settings;
  - чтение `.env`/переменных окружения;
  - настройка логирования.
- **API / веб-слой**:
  - FastAPI-приложение:
    - endpoint вебхука для Telegram;
    - health-check;
    - задел под будущие админские эндпоинты.

## Формат расписания и дневных лимитов

- **Рабочие окна**:
  - для каждого мастера задаётся список окон (`WorkingWindow`) по дням недели;
  - каждое окно: `weekday` (0–6, где 0 = Пн), `start_time`, `end_time` в локальном времени мастера;
  - пример: `Пн 10:00–12:00`, `Пн 12:00–14:00`.

- **Дневные лимиты по дням недели**:
  - для каждого дня недели задаётся `max_bookings`;
  - пример: `Пн – 3`, `Вт – 4`, `Ср – 0` (не принимаю в среду).
  - при достижении лимита день перестаёт считаться доступным для записи, даже если окна ещё есть.

- **Часовые пояса**:
  - для каждого мастера сохраняется таймзона (по умолчанию `Europe/Moscow`);
  - слоты и заявки хранятся в БД в UTC, отображаются в локальном времени мастера.

## LLM-парсинг прайс-листа

Парсер прайс-листа реализован через абстракцию:

- интерфейс `PriceListParser`:
  - метод `parse(text: str) -> PriceListParseResult`;
  - возвращает список услуг (`ParsedService`) с названием, ценой и (по возможности) длительностью.
- строгая валидация результата:
  - Pydantic-схемы;
  - проверка, что имя услуги не пустое;
  - цена > 0 и в разумных пределах;
  - длительность (если есть) > 0 и не чрезмерная;
  - ограничение на количество услуг.
- реализации:
  - `MockPriceListParser` — детерминированный мок для разработки и тестов;
  - опциональный OpenAI-совместимый адаптер, полностью изолированный настройками окружения.
- обработка изображений:
  - абстракция `ImageToTextAdapter`;
  - дефолтная реализация — заглушка, предлагающая мастеру прислать прайс текстом.
- при любой ошибке или низкой уверенности:
  - бот предлагает мастеру отредактировать/завести услуги вручную.

## Настройка и запуск локально

### 1. Подготовка окружения

1. Установите Python 3.12.
2. Установите Poetry:

```bash
pip install poetry
```

3. Склонируйте репозиторий:

```bash
git clone https://github.com/emris-git/nail-app.git
cd nail-app
```

4. Создайте `.env` на основе `.env.example` и заполните:
   - `TELEGRAM_BOT_TOKEN`
   - `DATABASE_URL` (для локальной разработки можно использовать docker-compose)
   - `WEBHOOK_BASE_URL` (необязательно для режима polling)
   - `BOT_WEBHOOK_SECRET`

### 2. Установка зависимостей

```bash
poetry install
```

### 3. Запуск БД и приложения через docker-compose

```bash
docker-compose up --build
```

Приложение будет доступно на `http://localhost:8000`.

### 4. Запуск только приложения (без Docker)

1. Поднимите PostgreSQL самостоятельно и установите `DATABASE_URL` в `.env`.
2. Выполните миграции:

```bash
poetry run alembic upgrade head
```

3. Запустите FastAPI+webhook-сервер:

```bash
make dev
```

или напрямую:

```bash
poetry run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 5. Запуск бота в режиме polling (для отладки)

```bash
make bot-polling
```

## Деплой на Railway (webhook)

1. Создайте новый проект на Railway и подключите репозиторий.
2. Добавьте переменные окружения:
   - `TELEGRAM_BOT_TOKEN`
   - `DATABASE_URL` (PostgreSQL-плагин Railway или внешний кластер)
   - `WEBHOOK_BASE_URL` (например, `https://<your-app>.up.railway.app`)
   - `BOT_WEBHOOK_SECRET` (случайная строка)
   - `DEFAULT_TIMEZONE` (например, `Europe/Moscow`)
3. Убедитесь, что переменная `PORT` установлена Railway (по умолчанию).
4. После деплоя:
   - установите webhook для Telegram, указав URL вида  
     `https://<your-app>.up.railway.app/webhook/<BOT_WEBHOOK_SECRET>`.

## Переменные окружения

Основные переменные (также перечислены в `.env.example`):

- `TELEGRAM_BOT_TOKEN` — токен бота от BotFather.
- `DATABASE_URL` — строка подключения к БД (PostgreSQL).
- `WEBHOOK_BASE_URL` — базовый URL для вебхука (Railway/хостинг).
- `BOT_WEBHOOK_SECRET` — секретный фрагмент пути вебхука.
- `DEFAULT_TIMEZONE` — таймзона по умолчанию (например, `Europe/Moscow`).
- `LLM_PROVIDER` — провайдер парсера прайса (`mock`, `openai`, ...).
- `LLM_API_BASE`, `LLM_API_KEY`, `LLM_MODEL` — параметры для LLM-провайдера (опционально).
- `LOG_LEVEL` — уровень логирования (`INFO`, `DEBUG` и т.п.).
- `PORT` — порт HTTP-сервера (на Railway задаётся автоматически).

## Конфигурация LLM-парсера

- По умолчанию для разработки и тестов используется `MockPriceListParser`.
- Для подключения внешнего LLM:
  - установите `LLM_PROVIDER=openai` (или другой поддерживаемый);
  - задайте `LLM_API_BASE`, `LLM_API_KEY`, `LLM_MODEL`;
  - убедитесь, что эти значения не попадают в репозиторий (хранятся только в `.env` или в переменных окружения хостинга).

Вся работа с LLM изолирована в адаптере, чтобы:

- можно было легко заменить провайдера;
- не ломать OSS-пользователей без платного LLM.
